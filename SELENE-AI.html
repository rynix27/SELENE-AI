The problem

Radar alone lies about ice.

Circular Polarization Ratio (CPR) above 1.0 is the textbook ice indicator — but rough terrain scatters radar the same way. Naive CPR thresholding paints the entire crater rim as "ice." SELENE-AI adds m-chi decomposition and a permanently-shadowed-region boundary check before calling anything a detection.

Stage 5 · Rover traverse

Minimum-energy path, not shortest path.

A shortest-distance path can cross a steep, rough slope that drains a solar-powered rover faster than a longer, gentler one. SELENE-AI costs every edge in the traverse graph accordingly, then runs Dijkstra’s algorithm to the target ice zone.

cost(edge) = distance × (1 + 0.05 × slope²) × roughness

Synthetic scene, real pipeline

  • Chandrayaan-2 DFSAR/OHRC access requires an ISSDC PRADAN account, which this environment does not have. Every grid on this page (CPR, DOP, slope, roughness, illumination, ice likelihood) comes from a physically-plausible synthetic doubly-shadowed crater scene, generated by data_simulator.py.
  • Every number after that point is computed, not authored — the feature extraction, Random Forest ice estimation, mission scoring, rover path planning and validation checks all ran for real against this synthetic scene, and this page reflects their actual output.
  • Swapping in real DFSAR/OHRC rasters means writing one loader that returns the same field names; every downstream stage is unchanged.
  • The one validation check that fails (spatial overlap with the reference ice zone, 50.6% against a >70% target) is left failing on this page rather than tuned away — it reflects a real precision/recall trade-off in the triple-evidence detector worth flagging for mission planners, not a bug.
